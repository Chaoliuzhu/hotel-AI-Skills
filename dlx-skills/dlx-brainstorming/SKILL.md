> 德胧 AI 龙虾军团核心技能 · 基于 brainstorming-design-workflow 深度定制

## 核心定位

在进入任何创意工作之前，**强制激活本 skill**。

本 skill 在标准头脑风暴流程（理解意图 → 探索方案 → 呈现设计）基础上，
**注入德胧决策框架**，确保每个想法都经得起"OCC、成本、C端、大佬"四重检验。

---

## 德胧决策四问（每个决策必答）

在开始任何方案讨论之前，**先问清楚这四点**：

| # | 问题 | 检查什么 |
|---|------|----------|
| 1 | **这个决策对 OCC 有什么帮助？** | OCC（出租率）是酒店生死线，任何方案必须能证明对 OCC 的拉动作用 |
| 2 | **花多少钱？多久回本？** | 成本纪律——每一分钱都要算清ROI，德胧不是烧钱的公司 |
| 3 | **谁是 C 端？C 端是否受益？** | C端优先论——客人、业主、员工，谁真正获益？ |
| 4 | **大佬会怎么看？（可选）** | 调用大佬角色卡，让张琦/毛新凯/李和聪参与决策 |

> ⚠️ 如果一个问题答不上来，**暂停**，先回答这个问题再继续。

---

## 决策标准速查

### OCC 优先

```
OCC = 出租率 = 酒店的生命线
- OCC 爬坡数据是最重要的验证指标
- 虫洞 AI 的核心价值：让客人"开了新能源车再开回油车"
- 温州龙港案例：OCC 48% → 83%+
```

### 成本纪律（毛新凯视角）

```
决策链五问：
1. 先看什么？财务回报 + ROI + C端价值
2. 排除什么？无法量化ROI的方案
3. 判断标准？花多少钱/赚多少/多久回本/C端受益
4. 何时改变？数据证明ROI不达预期
5. 核心底线？财务纪律不可破，信托责任

标志性语录：
"把公司的每一分钱当自己的钱，你才能知道该怎么花"
"如果这家店是你自己投资的店，你会怎么做？"
```

### C端优先论

```
C端定义（按优先级）：
1. 住客（Guest）—— 体验是否提升？
2. 业主（Owner）—— 投资回报是否增加？
3. 员工（Staff）—— 是否更轻松、更有获得感？

判断标准：
- 客户获得感 + 员工轻松 + OCC提升 = 三位一体
- B端收益是结果，不是出发点
```

### 大佬视角（角色卡调用）

当需要大佬参与时，从以下角色选择：

| 角色 | 置信度 | 适用场景 |
|------|--------|----------|
| **张琦** | 95% | 虫洞/AI方向、OCC爬坡、服务工具论 |
| **毛新凯** | 85% | 财务/成本/ROI、C端价值评估 |
| **李和聪** | 78% | 会员/532战略、数据验证 |

## 执行流程

### 阶段1：理解意图（必做）

**目标**：搞清楚用户真正想解决什么问题

**操作步骤**：
1. 查看项目现状（相关文件、最近提交）
2. 用德胧四问检验原始想法
3. 一次只问一个问题，优先封闭式问题
4. 聚焦：目的、约束、成功标准

**德胧四问检查表**：
```
原始想法：[用户的想法]

□ OCC关联：这个决策对OCC有什么帮助？
□ 成本纪律：花多少钱？多久回本？
□ C端优先：谁是C端？C端是否受益？
□ 大佬视角：需要调用哪位大佬？（张琦/毛新凯/李和聪/暂不需要）
```

### 阶段2：探索方案（必做）

**目标**：提出2-3个方案，带清楚trade-off

**操作步骤**：
1. 提出2-3个方案，每个方案必须回答德胧四问
2. 用表格对比，标注推荐方案
3. 说明推荐理由

**方案对比模板**：
```
| 维度     | 方案A（推荐） | 方案B | 方案C |
|----------|--------------|-------|-------|
| OCC影响  |              |       |       |
| 成本     |              |       |       |
| 回本周期 |              |       |       |
| C端受益  |              |       |       |
| 大佬评价 |              |       |       |
```

### 阶段3：呈现设计（分块）

**目标**：把方案拆成200-300字的小块，逐块确认

**模块顺序**：
1. 整体架构
2. 核心流程
3. 数据流
4. 错误处理
5. 测试策略

**每块结构**：
```
## [模块名]

[200-300字描述]

✓ 请确认：是否符合预期？需要调整什么？
```

### 阶段4：大佬审核（可选）

**触发条件**：用户说"大佬看看"、需要补充大佬视角、方案有争议

**操作步骤**：
1. 确定需要调用的角色（张琦/毛新凯/李和聪）
2. 读取dalao skill中的对应角色卡
3. 按角色决策链输出模拟意见
4. 标注置信度

**大佬审核输出格式**：
```
## 大佬视角审核

### 张琦视角（置信度95%）
[基于张琦决策链的方案评价]

### 毛新凯视角（置信度85%）
[基于毛新凯决策链的财务评价]

### 李和聪视角（置信度78%）
[基于李和聪决策链的数据验证]

> ⚠️ 以上均为AI推演，不代表真实人物立场
```

## 完整执行示例

### 示例：虫洞推广节奏头脑风暴

**用户输入**：
```
我们计划下季度在10家门店推广虫洞AI服务大使，
怎么安排推广节奏比较合理？
```

**阶段1：德胧四问确认**
```
✓ OCC关联：虫洞对OCC有拉动（参考APP保有率99.6%）
✓ 成本纪律：需要测算10家门店的总投入和回本周期
✓ C端优先：住客体验提升是核心价值
✓ 大佬视角：建议调用张琦（推广节奏）+ 毛新凯（成本）

原始想法补充：
- 是一次性推开还是分批？
- 每批几家？
- 评估指标是什么？
```

**阶段2：方案对比**
```
| 维度     | 方案A：快速扩张（推荐） | 方案B：循序渐进 | 方案C：标杆先行 |
|----------|------------------------|----------------|----------------|
| 节奏     | 2个月内10家全推         | 每批2家，跨半年 | 先1家，3个月后再复制 |
| OCC风险  | 中等（可接受）          | 低             | 低             |
| 总成本   | ~120万                 | ~150万         | ~80万          |
| 回本周期 | 6-8个月                | 8-10个月       | 4-6个月       |
| C端价值  | 快速规模化体验         | 稳扎稳打       | 极致打磨       |
| 张琦评价 | ⚠️ 风险与机遇并存      | ✓ 符合服务工具论 | ✓ 服务先行     |
| 毛新凯评价 | ✓ 可接受ROI          | ⚠️ 成本偏高     | ✓ 成本最优     |

**推荐**：方案A（快速扩张）
理由：虫洞已验证成熟（APP保有率99.6%），应快速规模化
```

**阶段3：设计呈现**（分块输出，每次200-300字）

**第一块：整体架构**
```
## 推广架构

采用"中心辐射+分批实施"模型：
- 总部：制定SOP标准、培训教材、监控看板
- 首批（3家）：选取OCC基础好的门店，1个月验证
- 扩量（7家）：首批验证后，2周内完成部署

每家门店部署周期：7天
- Day 1-2：硬件安装 + 网络调试
- Day 3-4：员工培训（服务大使）
- Day 5-6：试运行 + 问题修复
- Day 7：正式上线 + 数据监控

✓ 请确认：这个架构是否符合预期？
```

**第二块：评估指标**
```
## 核心评估指标

**OCC指标**（必须项）：
- OCC提升率：对比推广前后同期
- APP活跃率：目标 >90%
- 虫洞使用率：目标 >80%住客使用

**财务指标**（必须项）：
- 单店月均成本：AI工具费用 + 培训成本
- 回本周期：按OCC提升带来的增量收益计算
- ROI：目标 >150%（12个月）

**C端体验指标**（必须项）：
- 住客满意度：NPS提升
- 投诉率：下降目标20%
- 员工获得感：培训满意度 >90%

✓ 请确认：这些指标是否覆盖了德胧四问的要求？
```

**阶段4：毛新凯财务审核**
```
## 大佬视角审核

### 毛新凯视角（置信度85%）

**成本纪律审查**：
- 10家门店总投入：硬件（~30万）+ 培训（~40万）+ 运营（~50万）= ~120万
- 回本测算：OCC每提升1%，10家门店月均增收约15万
- 回本周期：6-8个月（可接受）

**C端价值确认**：
- 住客体验提升 → OCC提升 → 增收（逻辑成立）
- 符合"如果这家店是你自己投资的店，你会怎么做？"的检验

**风险提示**：
- ⚠️ 首批3家验证期是关键，不能跳过
- ⚠️ 建议设置"熔断机制"：若首批OCC提升 <2%，暂停扩量

**总体评价**：✓ 方案可行，但需严格执行分批验证

> ⚠️ 以上为AI推演，不代表真实人物立场
```

## 容错机制

### 异常处理

- **用户需求极度模糊，无法确定德胧四问答案**：先输出"德胧四问确认模板"，逐个问题引导用户聚焦；每个问题单独问，不一次抛全部四问
- **方案探索阶段无法产出多个可行方案**：调用dlx-copywriting生成3个备选思路作为思维启动；或搜索竞品相似方案作为参考基准
- **大佬角色卡文件不存在或读取失败**：跳过大佬审核环节，不阻断主流程；输出时标注「⚠️ 大佬视角未获取，建议人工确认」
- **方案对比表过于复杂**：自动精简对比维度至3个核心维度（OCC影响/成本/C端价值）；其余维度折叠为备注
- **文档输出失败（目录无写权限）**：改为输出到用户可访问的默认位置（如飞书文档）；标注「已保存至飞书，请查收」

### 决策边界

| 情况 | 决策 |
|------|------|
| 德胧四问中有任何一问无法回答 | **暂停**，优先回答该问题再继续，禁止跳过 |
| 方案数量 < 2 | 强制要求至少提出2个方案，否则不进入下一阶段 |
| 方案被用户否决 | 记录否决原因，返回阶段2重新探索，不直接结束 |
| 大佬意见与用户意见冲突 | 以用户意见为准，大佬意见标注置信度供参考 |
| 时间/进度压力 | 优先完成阶段1和阶段2，阶段4（大佬审核）可跳过 |

### 检查点机制

阶段完成时输出状态快照：`📍 [阶段N] 完成 | 核心问题：[已确认/待确认] | 方案数：X个 | 推荐方案：XXX`
